# Truth Spec 方案 Critical Audit

## 总体评价

Truth Spec 方案相比 v3 的源码推导是正确的方向，但当前设计存在 **15 个问题**，按严重程度分三级。

---

## 🔴 Critical（会导致假阴性/假阳性，必须修）

### C1. check_upstream_visibility 对 A4 的检查有逻辑漏洞

```python
# 当前代码:
for agent in expected_upstream:        # reviewing 的上游 = [A1, A2, A3, A4]
    mapping = context_map.get(agent, {})  # agent="A4" → 只取到 {source: "spec.openapi", context_path: "openapi_hint"}
    if not mapping:
        mapping = context_map.get(f"{agent}_erd", {})  # 这个 fallback 不会触发，因为 mapping 不为空
```

A4 对应**两个** context key（`openapi_hint` + `erd_hint`），但循环只检查了 `openapi_hint`。如果 openapi 正常但 erd 为空，这个检查不会报错——漏掉了 50% 的 A4 数据。

**修法：** context_key_mapping 改用一对多结构，或为 A4 单独展开：
```yaml
context_key_mapping:
  A4:
    - { source: "spec.openapi", context_path: "openapi_hint" }
    - { source: "spec.erd", context_path: "erd_hint" }
```

### C2. Gate 定义与实际代码行为不匹配

Truth Spec 写 `after_state: ANALYZING`，但实际代码中 Gate 0 是在 ANALYZING 状态**内部**执行的（先跑 A1，再跑 Gate，再转换到 DESIGNING）。

这导致一个歧义：`after_state` 究竟指"Gate 在哪个状态之后出现"还是"Gate 与哪个状态关联"？如果是前者，那么 Gate 审批成功后期待的状态转换应该是 `ANALYZING → DESIGNING`（由 `_compute_next_state` 计算）。但 check_gate_progression 用的是 `after_state` 来判断"审批后应该推进到哪个状态"——这里 `after_state` 被当成了"下一个状态"来用。

实际代码中 Gate 0 之后下一状态是 DESIGNING（由 `_linear_next` 映射），不是 ANALYZING。

**修法：** 明确 Gate 的 `next_state` 字段：
```yaml
gates:
  - level: 0
    runs_in_state: ANALYZING    # Gate 在哪个状态中执行
    next_state: DESIGNING        # 审批通过后进入哪个状态
```

### C3. 运行时可观测性有根本限制

Observer 通过监听 NATS 获取 context_package 和 agent result。但当前 NATS dispatch envelope（`dispatch_agent.py`）中的 payload 字段包含完整 context。需要确认：

1. NATS 消息的 payload 是否真的包含 `artifact_context`、`openapi_hint` 等完整嵌套结构？
2. `context.ready.>` 主题的消息是否一定被 observer 收到（JetStream 的 interest-based retention 意味着如果没有活跃 consumer，消息不会保留）？

如果 NATS 消息体不包括完整 context（例如被截断），那么 runtime_verifier 对 context 的所有检查都无法执行。这是一个**实现可行性**问题，必须在开始编码前验证。

**Action:** 先在服务器上抓一条真实的 `context.ready.spec_writer` 消息，确认 payload 结构。

## 🟠 Major（设计缺陷，影响覆盖范围）

### M1. 基础设施配置混入了 Truth Spec，造成误报

```yaml
infrastructure:
  llm:
    expected_model: deepseek-v4-pro-202606  # ← 这不是"预期行为"，这是"当前配置"
```

模型升级、数据库迁移到新端口、增加新的 NATS subject——这些都是运维变更，不是业务逻辑 bug。放在 Truth Spec 里会导致每次运维变更都触发"Spec 不一致"警告，团队会渐渐忽略这些警告（狼来了效应）。

**修法：** 把基础设施拆分出去：
- `truth-spec.yaml` — 只放业务逻辑契约（状态机、数据流、流程契约）
- `infra-baseline.yaml` — 基础设施基线（可选，首次部署时自动生成并提交 git，后续变更时自动提示更新）

### M2. 没有覆盖 A3/A4 并行不对称失败

`requirement_workflow.py:256-264`：A3 超时是 warning（非致命），A4 超时是 error（严重）。当前的 Truth Spec 只有：
```yaml
parallel_states:
  DESIGNING: [A3, A4]
```
完全丢失了 A3 vs A4 的失败语义差异。

**修法：**
```yaml
parallel_states:
  DESIGNING:
    agents: [A3, A4]
    failure_policy:
      A3: warn       # A3 超时不阻塞流程
      A4: error      # A4 超时需要处理
```

### M3. Truth Spec 的约束粒度无法处理 LLM 输出可变性

```yaml
a4_min_erd_tables: 1
a6_min_dag_nodes: 5
a9_max_iterations: 3
```

"至少 5 个 DAG 节点"对复杂需求是对的，但对 "加一个 health check 端点" 这种简单需求，LLM 可能只产出 3 个节点——而且是合理的。硬编码数值会制造假阳性。

**修法：** 区分"硬约束"和"软约束"：
- 硬约束（必须满足）：字段存在性、类型、状态值
- 软约束（通常满足，但例外也可接受）：数量下限。软约束失败时标 warning 而非 error。

### M4. NATS-Temporal Bridge 没有被显式验证

Bridge 是整个系统最脆弱的一环——它崩了，所有 Agent 结果都无法回传给 Temporal，工作流死锁。当前 spec 没有针对 Bridge 的检查。虽然 observer 通过对比 "agent.result 已发布" vs "Temporal 状态已推进" 可以间接发现，但没有显式的 Bridge 健康检查。

**修法：** 在 Layer 1 基础设施检查中增加 Bridge 探活：
- 发布测试消息到 `agent.result.test` → 确认测试 Temporal workflow 收到 Signal

### M5. Stub Agent（A7, A10, A11）没有特殊处理

A11 是 `A11TestAgentStub`，返回的是硬编码数据。如果 Truth Spec 要求 A11 输出特定字段，但 stub 返回的不满足要求，测试会永远失败——且无法修复（因为 stub 就是 stub）。

**修法：** Truth Spec 中标记 stub agent：
```yaml
agents:
  stubs: [A7, A10, A11]  # stub agent 仅检查 status=completed，不检查输出字段
```

### M6. 缺少数据写入的交叉表验证

A4 写入**三个地方**：`requirements.spec.openapi`、`api_schemas` 表、`erd_designs` 表。当前 Truth Spec 只跟踪 `spec.openapi` 和 `spec.erd`，没有提到独立表。

如果 `_write_spec_to_db` 成功了但 `_save_api_schema` 失败了（它们在不同的 try/catch 中），Truth Spec 无法发现。

**修法：**
```yaml
data_flow:
  persistence_contracts:
    A4:
      - target: spec.openapi           # JSONB 字段
      - target: api_schemas            # 独立表
      - target: spec.erd               # JSONB 字段
      - target: erd_designs            # 独立表
```

## 🟡 Minor（实现细节问题，可后续迭代修复）

### m1. Truth Spec 自洽性检查不够完善

当前只检查了 4 项。缺少：
- `context_key_mapping` 中引用的 `source` 路径是否存在于 DB schema 中
- `flow_contracts` 中的 `expect_in_context` 关键词是否可能在代码中实际出现（纯字符串匹配的可靠性验证）
- Gate `sla_hours` + `grace_hours` 之和不会导致无限等待

### m2. check_gate_progression 有时序假设

```python
idx = timeline.index(approval)
next_changes = [e for e in timeline[idx+1:] if ...]
```

如果 observer 的三个通道（NATS/DB/Temporal）由于网络延迟导致事件到达顺序不对，`timeline.index(approval)` 可能找不到正确的审批事件，或者下一个状态变更事件还没写入时间线。这个检查在时序不严格对齐时会误报。

### m3. Observer 的 DB 轮询间隔 (3s) 可能漏掉快速的状态转换

如果某个状态转换在 <3s 内完成（如 DRAFT → ANALYZING），DB 轮询可能直接从 DRAFT 跳到 ANALYZING，错过中间的快照。不过这种快速转换通常不重要（中间没有 Agent 执行），所以实际影响小。

### m4. `check_upstream_visibility` 中的"空值判断"可能误判

```python
is_empty = actual is None or (isinstance(actual, dict) and not actual) \
           or (isinstance(actual, list) and not actual)
```

一个包含全是空字符串值的 dict（如 `{"paths": {"": []}}`）会被判定为"非空"从而跳过检查，但实际上可能没有有效数据。这种边缘情况很少见，但理论存在。

### m5. Truth Spec 本身没有版本兼容性

当 Truth Spec 格式升级（如 `context_key_mapping` 从一对一改为一对多），旧格式的 Truth Spec 加载会报错。需要加一个 `meta.version` 字段和对应的迁移逻辑，但当前 spec 没提到。

### m6. 页面展示时，Truth Spec 推导的差异没有可视化

当"源码不一致"警告出现时，页面上如何展示？是弹窗、红色横幅、还是忽略？spec 中的页面设计没有为 Truth Spec 的三个检查结果预留展示区域。

## 🟢 建议补充的检查

以下场景当前 spec 完全没有覆盖，建议加：

1. **Agent 超时检测** — `_AGENT_TIMEOUTS` 定义了每个 Agent 的超时时间，测试工具应该在 Agent 接近超时时提前告警
2. **重复 dispatch 检测** — BASE worker 有 5 分钟去重窗口，超出后会重复执行。应检查同一 req_id+state 是否被 dispatch 了多次
3. **LLM 审计日志完整性** — 每个 Agent 执行后应该有对应的 LLM 调用记录，空记录说明 LLM 调用失败或无日志
4. **A9 worktree 泄漏** — 检查 `/tmp/a9-runtimes/` 是否有残留的 worktree 目录（`runtime.cleanup()` 没执行的情况）
5. **Gate 审批超时** — 如果 `gate_strategy=manual` 且审批时间超过 SLA，应该有明确提醒

## 📊 总结

| 级别 | 数量 | 核心问题 |
|------|------|----------|
| Critical | 3 | A4 数据检查不全、Gate 语义歧义、NATS 消息可观测性未验证 |
| Major | 6 | 基础设施误报、并行失败缺失、LLM 输出刚性约束、Bridge 盲区、Stub 处理、交叉表验证 |
| Minor | 6 | 自洽性不完整、时序假设、轮询间隔、空值判断、版本兼容、UI 缺失 |
| 建议补充 | 5 | 超时、重复 dispatch、审计日志、worktree 泄漏、Gate SLA |

**最紧迫的修复（必须在编码前解决）：**
1. C1 — A4 的 context_key_mapping 改为一对多
2. C2 — Gate 增加 `next_state` 字段，与 `after_state/runs_in_state` 区分
3. C3 — 先抓包验证 NATS 消息能不能提供足够数据给 runtime_verifier
4. M1 — 基础设施配置从 Truth Spec 中拆分出去
