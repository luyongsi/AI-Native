## 三、A6 完整设计 — Spec Decomposer

### 3.1 职责定位

A6 负责将阶段二产出的终版 Spec 拆解为可执行的任务 DAG，为后续 A9 开发提供任务分配依据。

### 3.2 输入

| 数据项 | 来源 | 说明 |
|--------|------|------|
| `spec_doc` | `agent_results.A4` → `design_specs` | 结构化 Spec 文档（模块/状态机/数据模型） |
| `openapi_schema` | `agent_results.A4` → `design_specs` | OpenAPI 3.0 Schema（API 端点定义） |
| `erd_diagram` | `agent_results.A4` → `design_specs` | ERD 实体关系图 |
| `ddl_statements` | `agent_results.A4` → `design_specs` | DDL SQL 语句 |
| `revision_context` | `context.ready.A6` payload | Gate2 打回时携带拒绝原因 + A8 报告 |

### 3.3 执行流程

```
1. 解析 context
   ├── 提取 spec_package (spec_doc + openapi_schema + erd_diagram + ddl_statements)
   └── 检查 revision_context.is_revision → 决定 prompt 中是否注入修正指引

2. LLM 拆解 (主路径)
   ├── 组装 prompt (见 §3.4)
   ├── 调用 DeepSeek API (temperature=0.2, max_tokens=4000)
   ├── 解析 JSON 响应
   └── 验证 DAG 结构 (见 §3.5)

3. Fallback 拆解 (备路径，LLM 失败时)
   ├── 基于 Spec 关键词分类 (has_backend / has_frontend / has_db)
   ├── 生成 5-8 个默认节点
   └── 标记 source='fallback'

4. 持久化
   ├── INSERT INTO task_dags (返回 dag_id)
   ├── UPSERT INTO agent_results (agent_key='A6')
   └── 发布 NATS: agent.result.A6 + dag.created
```

### 3.5 DAG 验证逻辑

```python
# 伪代码
def validate_dag(dag: dict) -> ValidationResult:
    nodes = dag.get("nodes", [])
    edges = dag.get("edges", [])
    node_ids = {n["id"] for n in nodes}

    errors = []
    warnings = []

    # 节点数检查
    if len(nodes) < 5:
        errors.append("节点数 < 5")
    elif len(nodes) > 25:
        errors.append("节点数 > 25，拒绝写入")
        return ValidationResult(valid=False, errors=errors)

    # 边有效性检查
    for e in edges:
        if e["from"] not in node_ids:
            warnings.append(f"边 {e['from']}→{e['to']}: from 引用无效节点")
        if e["to"] not in node_ids:
            warnings.append(f"边 {e['from']}→{e['to']}: to 引用无效节点")
        if e["from"] == e["to"]:
            errors.append(f"自环边: {e['from']}→{e['to']}")

    # 孤岛节点检查
    connected = set()
    for e in edges:
        connected.add(e["from"])
        connected.add(e["to"])
    isolated = node_ids - connected
    if isolated:
        warnings.append(f"孤岛节点（无边连接）: {isolated}")

    # critical_path 有效性检查
    cp = dag.get("critical_path", [])
    for nid in cp:
        if nid not in node_ids:
            warnings.append(f"critical_path 引用无效节点: {nid}")

    # 类型枚举检查
    valid_types = {"planning", "backend", "frontend", "db", "testing", "deployment"}
    valid_agents = {"A9", "A10", "A11", "A12"}
    for n in nodes:
        if n.get("type") not in valid_types:
            errors.append(f"节点 {n['id']}: 无效 type={n.get('type')}")
        if n.get("agent") not in valid_agents:
            errors.append(f"节点 {n['id']}: 无效 agent={n.get('agent')}")

    return ValidationResult(
        valid=len(errors) == 0,
        errors=errors,
        warnings=warnings,
    )
```

## 五、A8 完整设计 — Architecture Expert

### 5.1 职责定位

A8 是阶段三的**外部自审角色**（独立于 A6 的拆解流程），负责对 A6 产出的 DAG 进行静态分析 + LLM 架构评审，产出评分报告供 Gate2 审批人参考。

### 5.2 输入

| 数据项 | 来源 | 说明 |
|--------|------|------|
| `dag` (nodes + edges) | `context.ready.A8` payload / `task_dags` 表 | A6 产出的完整 DAG |
| `spec_doc` | `context.ready.A8` payload | Spec 文档（用于理解架构意图） |
| `openapi_schema` | `context.ready.A8` payload | API 契约（用于分层违规判断） |
| `erd_diagram` | `context.ready.A8` payload | 数据实体（用于 DB 回滚检查） |

### 5.3 执行流程

```
1. 接收 DAG (nodes + edges)
   ├── 从 context.ready.A8 或 dag.created 事件获取 DAG
   └── 若 DAG 为空（A6 超时降级）→ 跳过评审，标记 a8_missing

2. 阶段一：静态分析（纯算法，不依赖 LLM）
   ├── 循环依赖检测 (DFS 拓扑排序，多跳环)
   ├── 分层违规检测 (前端→DB 直连 / 跨服务边界异常)
   ├── DB 回滚检查 (db 类型节点的 steps 中是否含 rollback/回滚)
   └── 生成静态分析结果

3. 阶段二：LLM 架构评审
   ├── 组装 prompt (含 DAG 摘要 + 静态分析结果)
   ├── 调用 DeepSeek API (temperature=0.1, max_tokens=2000)
   ├── 检查项: 架构分层合理性、安全风险 (SEC-AUTH/SEC-SQL/SEC-DATA/SEC-KEY)
   │           性能风险 (PERF-N1/PERF-CACHE/PERF-PAGE)、模块耦合度
   └── 返回 violations + suggestions + summary

4. 阶段三：合并报告
   ├── 合并静态分析 violations + LLM violations
   ├── 计算总分 (评分规则见 §5.6)
   ├── 判定 verdict (score ≥ 70 且无循环依赖 → pass, 否则 fail)
   └── 生成最终报告

5. 持久化
   ├── UPSERT INTO agent_results (agent_key='A8')
   └── 发布 NATS: agent.result.A8
```

### 5.4 静态检查算法

#### 5.4.1 循环依赖检测（DFS 拓扑排序）

```python
def detect_cycles(nodes: list, edges: list) -> tuple[bool, list]:
    """使用 DFS 颜色标记法检测有向图中的环"""
    WHITE, GRAY, BLACK = 0, 1, 2

    graph = defaultdict(list)
    node_ids = {n["id"] for n in nodes}
    for e in edges:
        f, t = e["from"], e["to"]
        if f in node_ids and t in node_ids:
            graph[f].append(t)

    color = defaultdict(lambda: WHITE)
    parent = {}

    def dfs(u, path):
        color[u] = GRAY
        for v in graph.get(u, []):
            if color[v] == GRAY:
                # 找到环: 从 v 到 u 沿 parent 回溯
                cycle = [v]
                cur = u
                while cur != v:
                    cycle.append(cur)
                    cur = parent[cur]
                cycle.append(v)
                return cycle[::-1]
            elif color[v] == WHITE:
                parent[v] = u
                result = dfs(v, path + [v])
                if result:
                    return result
        color[u] = BLACK
        return None

    for nid in node_ids:
        if color[nid] == WHITE:
            cycle = dfs(nid, [nid])
            if cycle:
                return True, cycle

    return False, []
```

#### 5.4.2 分层违规检测

```python
# 合法分层: frontend → backend → db（单向依赖，不可反向/跨层）
_LAYER_ORDER = {"frontend": 0, "backend": 1, "db": 2}

def check_layer_violations(nodes: list, edges: list) -> list[dict]:
    violations = []
    node_map = {n["id"]: n for n in nodes}

    for e in edges:
        f = node_map.get(e["from"])
        t = node_map.get(e["to"])
        if not f or not t:
            continue

        # 前端直接依赖 DB → critical
        if f.get("type") == "frontend" and t.get("type") == "db":
            violations.append({
                "rule": "LAYER-VIO-001",
                "severity": "critical",
                "title": f"跨层调用: {f['id']}(frontend) → {t['id']}(db)",
                "detail": "前端节点直接依赖数据库层，应通过 API 接口（backend）间接访问",
                "suggestion": "在前端和 DB 之间插入 backend 节点，通过 REST API 或 RPC 调用",
                "affected_nodes": [f["id"], t["id"]],
            })

        # DB 依赖前端 → warning（不太可能但检查）
        if f.get("type") == "db" and t.get("type") == "frontend":
            violations.append({
                "rule": "LAYER-VIO-002",
                "severity": "warning",
                "title": f"反向依赖: {f['id']}(db) → {t['id']}(frontend)",
                "detail": "数据库节点不应依赖前端展示层",
                "suggestion": "解除反向依赖，数据流向应为 frontend → backend → db",
                "affected_nodes": [f["id"], t["id"]],
            })

    return violations
```

#### 5.4.3 DB 回滚检查

```python
def check_db_rollback(nodes: list) -> list[dict]:
    """检查 db 类型节点的 steps 中是否包含回滚/rollback 方案"""
    issues = []

    for n in nodes:
        if n.get("type") != "db":
            continue

        steps = n.get("steps", [])
        has_rollback = any(
            "rollback" in str(s).lower() or "回滚" in str(s)
            for s in steps
        )

        if not has_rollback:
            issues.append({
                "rule": "DB-ROLLBACK-001",
                "severity": "warning",
                "title": f"DB 节点 {n['id']} 缺少回滚方案",
                "detail": f"节点 '{n.get('title','')}' 的 steps 中未包含数据库迁移的回滚（down migration）方案",
                "suggestion": "为每个 DDL 变更补充反向迁移步骤（如 ALTER TABLE DROP COLUMN → 回滚: ALTER TABLE ADD COLUMN）",
                "affected_nodes": [n["id"]],
            })

    return issues
```

## 八、A6 ↔ A8 对抗循环设计（P1）

### 8.1 触发条件

```python
def should_trigger_antagonism(a8_result: dict) -> bool:
    score = a8_result.get("score", 0)
    revision_count = a8_result.get("stage3_revision_count", 0)
    return (
        50 <= score < 70
        and revision_count < 2
        and not a8_result.get("cycle_detected", False)
    )
```

### 8.2 执行流程

```
A8 评审完成
  → should_trigger_antagonism?
  ├── 是:
  │     → Orchestrator 发布 context.ready.A6（含 a8_suggestions）
  │     → task_dags.stage3_revision_count += 1（记录在下一版 DAG 中）
  │     → A6 接收建议，修正 DAG
  │     → A6 发布 agent.result.A6（修订版，同 cycle）
  │     → Orchestrator 重新发布 context.ready.A8
  │     → A8 重新评审
  │     → 若仍触发 → 重复（直至 2 轮上限）
  └── 否:
        ├── score < 50 或循环依赖 → 生成《架构分歧报告》→ Gate2 人工裁决
        ├── 达到 2 轮上限 → 生成《架构分歧报告》→ Gate2 人工裁决
        └── score ≥ 70 → 正常流转 Gate2
```

> **`stage3_revision_count` 语义**：该计数器追踪当前 Gate2 审批轮次内的 A6↔A8 对抗轮次（每轮 +1）。Gate2 打回后进入新 cycle，计数器**重置为 0**——即 `stage3_revision_count` 不跨 Gate2 审批周期累加。如需追溯全链路修订历史，使用 `(req_id, cycle, version)` 三元组和 `event_log`。

## 九、安全与性能考虑

### 9.1 安全检查（A8 强制执行）

## 十、错误处理与降级策略

| 场景 | 影响 | 降级策略 | 恢复方式 |
|------|------|---------|---------|
| **DeepSeek API 不可用** | A6/A7/A8 LLM 调用失败 | A6/A7 使用 fallback 规则模式；A8 仅出静态分析报告 | DeepSeek 恢复后自动切换回 LLM 模式 |
| **task_dags 写入失败** | DAG 无法持久化 | 重试 3 次（30s 间隔）→ 写入 agent_results 作为唯一记录 | 手动/自动从 agent_results 恢复写入 task_dags |
| **test_assets 写入失败** | 测试用例丢失 | 重试 3 次 → 写入 agent_results 作为兜底 | 手动/自动从 agent_results 恢复 |
| **NATS 投递失败** | Agent 结果通知丢失 | Outbox 重试 5 次（30s 间隔）→ 死信队列 | 从死信队列重放 |
| **A6 超时** | DAG 未产出 | 重试 1 次 → a6_missing=true → Gate2 审批人见空 DAG | Gate2 打回后重新触发 |
| **A7 超时** | 测试用例未产出 | 重试 1 次 → a7_missing=true → Gate2 审批人见缺失提示 | Gate2 打回后重新触发 |
| **A8 超时** | 架构评审报告缺失 | 重试 1 次 → a8_missing=true → Gate2 审批人仅看 DAG+用例 | Gate2 打回后重新触发 |
| **Gate2 审批人无响应** | 流程阻塞 | 2h 预警 → 4h 升级 → 持续等待 | 管理员手动指派替代审批人 |
