## 一、与阶段一/二的关系

### 1.1 扩展原则

| 原则 | 说明 |
|------|------|
| **增量扩展** | 不修改阶段一/二已定义的表字段语义，仅追加新列/新表/新枚举 |
| **复用表** | `requirements`、`agent_results`、`event_log`、`approvals` 四张核心表继续使用 |
| **新表** | 阶段三特有产物由 `task_dags` 承载（`test_assets` 表 A7 已在阶段二预建，阶段三规范化） |
| **全局路由键不变** | `req_id`、`session_id`、`cycle` 三键贯穿全链路 |

### 1.2 阶段衔接

```
Gate1 pass
  → requirements.phase = 'tech_prep'
  → requirements.tech_prep_status = 'decomposing'
  → requirements.tech_prep_revision_count = 0
  → Orchestrator 并行发布 context.ready.A6 + context.ready.A7
  → 阶段三开始
```

### 1.3 阶段三内部修订机制

与阶段一/二不同，阶段三的修订**不改变 `cycle`**，使用独立的 `tech_prep_revision_count` 追踪：

```
Gate2 reject
  → tech_prep_revision_count += 1
  → tech_prep_status = 'revising'
  → cycle 不变（同一 req 内循环）
  → A6 + A7 + A8 完整重走

Gate2 pass
  → phase = 'development'
  → tech_prep_status = 'tech_prep_completed'
  → tech_prep_revision_count 冻结
```

> `cycle` 仅在 Gate0 拒绝时递增（回归阶段一），表示需求层面的重新理解。阶段三内的 DAG/测试用例修订属于技术准备层面的迭代，不触发需求重新理解。

## 二、复用表的扩展

### 2.1 requirements 表扩展

阶段一/二已有列保持不变，新增以下列：

```sql
-- 新增：阶段三子状态
ALTER TABLE requirements ADD COLUMN tech_prep_status VARCHAR(30);
-- tech_prep_status 枚举: 'decomposing' | 'decomposed' | 'test_ready' | 'reviewing' | 'revising' | 'tech_prep_completed'

-- 新增：阶段三内部修订计数（不改变 cycle）
ALTER TABLE requirements ADD COLUMN tech_prep_revision_count INT DEFAULT 0;
```

**状态字段分工（全阶段）：**

| 字段 | 用途 | 更新者 |
|------|------|--------|
| `status` | 阶段一状态 | A1 / Orchestrator |
| `phase` | 当前所处阶段 | Orchestrator |
| `design_status` | 阶段二子状态 | Orchestrator / A3 confirm |
| `design_revision_count` | 阶段二修订计数 | Orchestrator（Gate1 拒绝时 +1） |
| `tech_prep_status` | 阶段三子状态 | Orchestrator |
| `tech_prep_revision_count` | 阶段三修订计数 | Orchestrator（Gate2 拒绝时 +1） |

**tech_prep_status 枚举：**

| 值 | 含义 | 设置者 | 触发条件 |
|----|------|--------|---------|
| `decomposing` | A6 + A7 正在执行 | Orchestrator | Gate1 pass / Gate2 reject 后重新调度 |
| `decomposed` | A6 完成（DAG 已产出） | Orchestrator | 收到 `agent.result.A6`（A7 仍在跑） |
| `test_ready` | A7 完成（测试用例已产出） | Orchestrator | 收到 `agent.result.A7`（A6 仍在跑） |
| `reviewing` | A8 评审中或等待 Gate2 审批 | Orchestrator | A6+A7 都完成后 |
| `revising` | Gate2 打回后 A6+A7 修订中 | Orchestrator | 收到 `agent.result.gate2.reject` |
| `tech_prep_completed` | Gate2 pass，阶段三完成 | Orchestrator | 收到 `agent.result.gate2.pass` |

**阶段三状态流转图：**

```
Gate1 pass
  → phase = 'tech_prep'
  → tech_prep_status = 'decomposing'
  → tech_prep_revision_count = 0

A6 完成 (agent.result.A6)
  ├── 若 A7 仍在执行 → tech_prep_status = 'decomposed'
  └── 若 A7 也已完成 → tech_prep_status = 'reviewing'

A7 完成 (agent.result.A7)
  ├── 若 A6 仍在执行 → tech_prep_status = 'test_ready'
  └── 若 A6 也已完成 → tech_prep_status = 'reviewing'

Gate2 pass (agent.result.gate2.pass)
  → phase = 'development'
  → tech_prep_status = 'tech_prep_completed'
  → tech_prep_revision_count 不再变化

Gate2 reject (agent.result.gate2.reject)
  → tech_prep_status = 'revising'
  → tech_prep_revision_count += 1
  → 重新调度 A6+A7 → tech_prep_status = 'decomposing'
```

> `decomposed` / `test_ready` 是**瞬时残差状态**，仅在 A6/A7 执行速度不同步时出现。正常场景下 Orchestrator 在收到第六条 `agent.result.A6/A7` 后立即转换为 `reviewing`。

### 2.2 agent_results 表扩展

表结构不变。新增 Agent Key 枚举值：

| agent_key | 说明 | status 语义 |
|-----------|------|------------|
| `'A6'` | Spec 拆解 Agent | `'completed'` / `'empty'`（LLM+fallback 均失败） |
| `'A7'` | 测试用例生成 Agent | `'completed'` / `'skipped'`（Orchestrator 超时跳过） |
| `'A8'` | 架构评审 Agent | `'completed'` / `'skipped'`（Orchestrator 超时跳过） |

> UNIQUE (req_id, agent_key, cycle) 约束不变。每轮 cycle 保留独立快照。
>
> **阶段内修订的写入策略**：Gate2 拒绝时 cycle 不变，A6/A7/A8 在**同一 cycle 内重新执行**时会再次写入 agent_results。为避免 UNIQUE 冲突，所有写入必须使用 **UPSERT**（`INSERT ... ON CONFLICT (req_id, agent_key, cycle) DO UPDATE SET artifact=EXCLUDED.artifact, status=EXCLUDED.status, created_at=NOW()`），而非纯 INSERT。此策略与阶段二 §2.2 一致。

### 2.3 approvals 表扩展

新增 gate_level=2 的行。新增两个布尔字段用于控制 Gate2 打回范围：

```sql
ALTER TABLE approvals ADD COLUMN a6_rework BOOLEAN DEFAULT true;
ALTER TABLE approvals ADD COLUMN a7_rework BOOLEAN DEFAULT true;
```

- `a6_rework`：Gate2 拒绝时是否需要 A6 返工（默认 true）
- `a7_rework`：Gate2 拒绝时是否需要 A7 返工（默认 true）
- 两字段分开控制，支持"仅需修订 DAG，测试用例可用"等场景
- 仅 gate_level=2 时有效，gate_level=0/1 时始终为 false

### 2.4 event_log 表

无需变更。

## 三、新增表

### 3.1 task_dags 表（A6 管理，Orchestrator 读取）

```sql
CREATE TABLE task_dags (
    id                  BIGSERIAL PRIMARY KEY,
    req_id              UUID NOT NULL REFERENCES requirements(id),
    cycle               INT NOT NULL DEFAULT 0,
    version             INT NOT NULL DEFAULT 1,        -- 修订版本号（A6↔A8 对抗或 Gate2 打回后递增）
    dag_json            JSONB NOT NULL,                 -- 完整 DAG 结构（见下方结构规范）
    node_count          INT,                            -- 节点总数
    critical_path_length INT,                           -- 关键路径长度（DAG 最长路径节点数）
    total_estimated_hours NUMERIC(6,1),                 -- 预估总工时
    human_review_nodes  INT DEFAULT 0,                  -- 标记为人工审核的节点数
    source              VARCHAR(20) DEFAULT 'llm',      -- 'llm' | 'llm_no_mcp' | 'fallback'
    stage3_revision_count INT DEFAULT 0,                -- A6↔A8 对抗循环计数（P1）
    created_at          TIMESTAMPTZ DEFAULT NOW(),
    updated_at          TIMESTAMPTZ DEFAULT NOW(),

    UNIQUE (req_id, cycle, version)
);

CREATE INDEX idx_task_dags_req ON task_dags(req_id, cycle, version DESC);
CREATE INDEX idx_task_dags_created ON task_dags(req_id, created_at DESC);
```

**dag_json 结构规范：**

```json
{
  "nodes": [
    {
      "id": "task-01",
      "type": "planning|backend|frontend|db|testing|deployment",
      "title": "任务标题（简明扼要）",
      "description": "任务详细描述",
      "complexity": "low|medium|high",
      "estimated_hours": 4,
      "agent": "A9|A10|A11|A12",
      "steps": ["步骤1", "步骤2"],
      "needs_human_review": false,
      "human_review_reason": null
    }
  ],
  "edges": [
    {"from": "task-01", "to": "task-02", "type": "sequential|parallel"}
  ],
  "critical_path": ["task-01", "task-03", "task-05"],
  "parallel_groups": [
    {"name": "前后端并行开发", "tasks": ["task-02", "task-04"]}
  ],
  "total_estimated_hours": 48,
  "metadata": {
    "node_count": 8,
    "human_review_nodes": 2,
    "high_complexity_nodes": 1,
    "source": "llm"
  }
}
```

**字段约束：**
- `nodes` 数组：最少 5 个，最多 20 个（A6 prompt 约束，代码层硬限制 ≤ 25）
- `edges` 数组：`from`/`to` 必须引用有效的 node.id
- `critical_path` 数组：所有元素必须存在于 nodes 中
- `parallel_groups[].tasks`：所有元素必须存在于 nodes 中
- `estimated_hours` 字段：正整数，单位小时
- `type` 仅允许 6 种枚举值
- `complexity` 仅允许 3 种枚举值
- `agent` 仅允许 `A9` / `A10` / `A11` / `A12`

**dag_json 验证规则（代码层）：**

| 规则 | 检查方式 | 失败动作 |
|------|---------|---------|
| 节点数量 5 ≤ N ≤ 25 | 计数 | 拒绝写入，返回错误 |
| 所有 edge.from/to 引用有效节点 | 集合查重 | 移除无效边，记录 warning |
| critical_path 元素有效 | 子集校验 | 移除无效路径节点 |
| parallel_groups 去重 | 集合查重 | 去重并记录 warning |
| 无自环边（from != to） | 逐条检查 | 移除自环边 |
| 至少一条顺序链路从 task-01 到最终任务 | 路径检查 | 记录 warning，不拒绝 |

### 3.2 test_assets 表（A7 管理，确认已有表结构）

A7 现有代码已向 `test_assets` 表写入结构化测试资产。阶段三据此确认正式结构：

```sql
CREATE TABLE test_assets (
    id                      BIGSERIAL PRIMARY KEY,
    req_id                  UUID NOT NULL REFERENCES requirements(id),
    unit_tests              JSONB,          -- 单元测试用例数组（每条含 title/type/priority/steps/tags/node_id）
    integration_tests       JSONB,          -- 集成测试用例数组
    e2e_tests               JSONB,          -- E2E 测试用例数组
    visual_tests            JSONB,          -- 可视化测试用例数组
    coverage_targets        JSONB,          -- 覆盖率目标 {overall, branches, lines}
    total_cases             INT,            -- 用例总数
    priority_distribution   JSONB,          -- {P0: n, P1: n, P2: n}
    source                  VARCHAR(50) DEFAULT 'a7_generator',
    version                 INT DEFAULT 1,
    created_at              TIMESTAMPTZ DEFAULT NOW(),
    updated_at              TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_test_assets_req ON test_assets(req_id, created_at DESC);
```

> **A7 写入策略**：`INSERT INTO test_assets (...) VALUES (...) RETURNING id`（不使用 UPSERT，每次写新记录）。版本递增由 `version = COALESCE((SELECT MAX(version) FROM test_assets WHERE req_id=?), 0) + 1` 实现。
>
> **A11 读取方式**：`SELECT * FROM test_assets WHERE req_id=? ORDER BY created_at DESC LIMIT 1` 获取最新测试资产。

**测试用例单条结构（unit_tests/integration_tests/e2e_tests/visual_tests 数组元素）：**

```json
{
  "title": "测试用例标题",
  "type": "unit|integration|e2e|visual|api",
  "priority": "P0|P1|P2",
  "description": "简要说明",
  "preconditions": "前置条件",
  "steps": [
    {"step_number": 1, "action": "操作步骤", "expected": "预期结果"}
  ],
  "tags": ["标签1", "标签2"],
  "node_id": "task-01"
}
```

| `PERF-CACHE-001` | 性能风险 | `warning` | LLM 分析 | 缺少缓存层 |
| `PERF-PAGE-001` | 性能风险 | `warning` | LLM 分析 | 未考虑分页 |

### 6.4 评分规则

| 条件 | 扣分 | 说明 |
|------|------|------|
| 检测到循环依赖 | -30 | 且直接判定 verdict='fail'，忽略 score 阈值 |
| 每个 critical violation | -10 | 安全/分层严重违规 |
| 每个 warning violation | -5 | DB 回滚/性能风险 |
| 基础分为 100，最低 0 | — | — |

**verdict 判定**：
- 循环依赖 → `fail`（强制）
- score ≥ 70 → `pass`
- score < 70 → `fail`

## 八、全链路事件流

### 8.1 正常流程

```
Gate1 pass → Orchestrator:
  → 写入 event_log (direction='IN', event_name='agent.result.gate1.pass')
  → 更新 requirements:
      - phase = 'tech_prep'
      - tech_prep_status = 'decomposing'
      - tech_prep_revision_count = 0
  → 查询 DB（MAX cycle）: requirements + agent_results (A1-A5) + design_specs
  → 并行发布 NATS:
      - context.ready.A6（build_context 含 spec_package + a1_output + a2_output + revision_context）
      - context.ready.A7（build_context 含 spec_package + dag_preview + revision_context）
  → 写入 event_log (direction='OUT', event_name='context.ready.A6')
  → 写入 event_log (direction='OUT', event_name='context.ready.A7')

A6 完成 (agent.result.A6):
  → 写入 event_log (direction='IN', event_name='agent.result.A6')
  → 更新 requirements.tech_prep_status:
      - 若 tech_prep_status != 'test_ready' → 'decomposed'
      - 若 tech_prep_status == 'test_ready' → 'reviewing'（GATHER）
  → 检查 GATHER: A6 + A7 都完成？→ 若都完成，执行 GATHER 步骤

A7 完成 (agent.result.A7):
  → 写入 event_log (direction='IN', event_name='agent.result.A7')
  → 更新 requirements.tech_prep_status:
      - 若 tech_prep_status != 'decomposed' → 'test_ready'
      - 若 tech_prep_status == 'decomposed' → 'reviewing'（GATHER）
  → 检查 GATHER: A6 + A7 都完成？→ 若都完成，执行 GATHER 步骤

GATHER (A6 + A7 都完成):
  → 更新 requirements.tech_prep_status = 'reviewing'
  → 查询 DB: dag (from tasks_dags 或 agent_results A6) + spec_package
  → build_context → 发布 NATS: context.ready.A8
  → 写入 event_log (direction='OUT', event_name='context.ready.A8')

A8 完成 (agent.result.A8):
  → 写入 event_log (direction='IN', event_name='agent.result.A8')
  → 检查是否需要 A6↔A8 对抗循环（P1-01）
      - score ∈ [50, 70) 且对抗轮次 < 2 → 触发对抗
      - 否则 → 继续 Gate2
  → 查询 DB: 全部 A1-A8 产物
  → build_context → 发布 NATS: context.ready.gate2
  → 写入 event_log (direction='OUT', event_name='context.ready.gate2')
  → MC Backend 预创建 approvals (gate_level=2, status='pending')

Gate2 pass (agent.result.gate2.pass):
  → 写入 event_log (direction='IN', event_name='agent.result.gate2.pass')
  → 更新 requirements:
      - phase = 'development'
      - tech_prep_status = 'tech_prep_completed'
  → 启动阶段四 → 发布 context.ready.A9
```

### 8.2 Gate2 打回链路

```
Gate2 reject (agent.result.gate2.reject):
  → 写入 event_log (direction='IN', event_name='agent.result.gate2.reject')
  → 更新 requirements:
      - tech_prep_status = 'revising'
      - tech_prep_revision_count += 1
  → 从 payload 读取 a6_rework / a7_rework 标记
  → 查询 DB: gate2_rejection context + 当前最新 DAG + test_assets
  → 创建 revision_context（含 gate2_rejection + previous_a8_report）
  → 根据 rework 标志并行发布:
      ├── a6_rework=true → 发布 context.ready.A6（含 revision_context）
      └── a7_rework=true → 发布 context.ready.A7（含 revision_context）
  → 写入 event_log (direction='OUT')
  → 修订后完整重走 A6+A7 → A8 → Gate2 链路（cycle 不变）
```

### 8.3 超时降级

```
A6 超时（10 分钟）:
  → Orchestrator 重试 1 次（重新发布 context.ready.A6）→ 仍超时
  → 写入 agent_results (agent_key='A6', cycle, status='empty')
  → 写入 task_dags (dag_json={}, source='timeout')
  → 设置 a6_missing = true
  → 继续 GATHER 逻辑（视为 A6 "完成"但空产出）

A7 超时（10 分钟）:
  → Orchestrator 重试 1 次 → 仍超时
  → 写入 agent_results (agent_key='A7', cycle, status='skipped')
  → 设置 a7_missing = true
  → 继续 GATHER 逻辑

A8 超时（10 分钟）:
  → Orchestrator 重试 1 次 → 仍超时
  → 写入 agent_results (agent_key='A8', cycle, status='skipped')
  → 设置 a8_missing = true
  → 继续 Gate2（无架构评审报告，审批人仅看 DAG + 测试用例）
```

> 超时阈值 `AGENT_TIMEOUT_SECONDS = 600`，可通过环境变量覆盖。

## 九、审计查询速查（阶段三）

| 审计问题 | 查询路径 |
|---------|---------|
| DAG 经历了几个版本？ | `SELECT MAX(version) FROM task_dags WHERE req_id=? AND cycle=?` |

| 阶段三耗时？ | `event_log WHERE req_id=? AND event_name IN ('context.ready.A6', 'agent.result.gate2.pass')` 首末 created_at 差值 |
| 超时降级记录？ | `agent_results WHERE req_id=? AND agent_key IN ('A6','A7','A8') AND status IN ('empty','skipped')` |
| A6↔A8 对抗进行了几轮？ | `task_dags WHERE req_id=? AND cycle=? → stage3_revision_count (MAX)` |
| Gate2 的完整审批记录？ | `approvals WHERE gate_level=2 AND req_id=? ORDER BY created_at DESC` |
