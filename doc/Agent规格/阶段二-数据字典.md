# 阶段二 数据字典

## 文档信息
- **版本**: v1.1
- **日期**: 2026-07-13
- **状态**: 阶段二（A3 / A4 / A5 / Gate1 / Orchestrator）全链路数据规范（已通过 critical 审计）
- **参考**: [系统状态机与信息流设计](../系统状态机与信息流设计.md) · [阶段一数据字典](./阶段一-数据字典.md)
- **说明**: 本文档是阶段二的**唯一数据规范源**。各 Agent/Gate/Orchestrator 设计文档中涉及的数据结构、字段名、存储位置、事件 payload，均以此文档为准。**本文档是阶段一数据字典的增量扩展，不重复定义阶段一已有的表结构和枚举。**

## 一、与阶段一的关系

### 1.1 扩展原则

| 原则 | 说明 |
|------|------|
| **增量扩展** | 不修改阶段一已定义的表字段语义，仅追加新列/新表/新枚举 |
| **复用表** | `requirements`、`agent_results`、`event_log`、`approvals` 四张核心表继续使用 |
| **新表** | 阶段二特有产物由 `prototype_artifacts`、`design_specs` 承载 |
| **全局路由键不变** | `req_id`、`session_id`、`cycle` 三键贯穿全链路 |

### 1.2 阶段衔接

```
Gate0 pass
  → requirements.phase = 'design', design_status = 'prototyping'
  → Orchestrator 发布 context.ready.A3
  → 阶段二开始
```

## 二、复用表的扩展

### 2.1 requirements 表扩展

阶段一已有列保持不变，新增以下列：

```sql
-- 新增：阶段标识
ALTER TABLE requirements ADD COLUMN phase VARCHAR(20) DEFAULT 'requirement';
-- phase 枚举: 'requirement' | 'design' | 'tech_prep' | 'development'

-- 新增：阶段二子状态
ALTER TABLE requirements ADD COLUMN design_status VARCHAR(30);
-- design_status 枚举: 'prototyping' | 'spec_writing' | 'design_checking' | 'design_completed' | 'gate1_rejected'

-- 新增：阶段二内部修订计数（A4 返工次数）
ALTER TABLE requirements ADD COLUMN design_revision_count INT DEFAULT 0;

-- 新增：A4 Spec 产物镜像（供 A5/Gate1 在 NATS 不可用时直接从 DB 读取兜底）
ALTER TABLE requirements ADD COLUMN spec JSONB DEFAULT '{}'::jsonb;
```

**状态字段分工：**

| 字段 | 用途 | 更新者 |
|------|------|--------|
| `status` | 阶段一状态（draft → analyzing_completed ↔ gate_rejected → approved） | A1 / Orchestrator |
| `phase` | 当前所处阶段 | Orchestrator |
| `design_status` | 阶段二内子状态 | Orchestrator / A3 confirm 时 |
| `design_revision_count` | A4 返工次数（阶段内，不改变 cycle） | Orchestrator（Gate1 拒绝时 +1） |

**阶段二状态流转：**

```
Gate0 pass
  → phase = 'design'
  → design_status = 'prototyping'

A3 用户确认原型
  → design_status = 'spec_writing'

A4 Spec 完成（agent.result.A4）
  → design_status = 'design_checking'

A5 设计检查完成（agent.result.A5）
  → design_status = 'design_completed'

Gate1 pass
  → phase = 'tech_prep'    （进入阶段三）
  → design_status = NULL   （阶段二结束）

Gate1 reject
  → design_status = 'gate1_rejected'（瞬时状态，Orchestrator 处理后立即转为 prototyping / spec_writing）
  → design_revision_count += 1
  ├── 一般拒绝 → Orchestrator 发布 context.ready.A4（A4 修订）
  │     → design_status = 'spec_writing'
  └── 含 a3_rework 标记 → Orchestrator 发布 context.ready.A3（A3 返工）
        → design_status = 'prototyping'

> `gate1_rejected` 是瞬时状态。Orchestrator 在收到 Gate1 拒绝后立即设置，随后根据 a3_rework 标志转换为 `spec_writing` 或 `prototyping`。仅在 Orchestrator 异常中断时此状态持续存在，可用于恢复判断。
```

### 2.2 agent_results 表扩展

表结构不变。新增 Agent Key 枚举值：

| agent_key | 说明 | status 语义 |
|-----------|------|------------|
| `'A3'` | UI 原型 Agent | `'completed'`（A3 始终用此值） |
| `'A4'` | Spec 撰写 Agent | `'completed'` / `'skipped'`（Orchestrator 超时跳过） |
| `'A5'` | 自动设计检查 Agent | `'completed'` / `'skipped'` |

> UNIQUE (req_id, agent_key, cycle) 约束不变。每轮 cycle 保留独立快照。
>
> **阶段内修订的写入策略**：Gate1 拒绝时 cycle 不变，A3/A4/A5 在**同一 cycle 内重新执行**时会再次写入 agent_results。为避免 UNIQUE 冲突，所有 agent_results 写入必须使用 **UPSERT**（`INSERT ... ON CONFLICT (req_id, agent_key, cycle) DO UPDATE SET artifact=EXCLUDED.artifact, status=EXCLUDED.status, created_at=NOW()`），而非纯 INSERT。此策略保证最新执行结果覆盖同 cycle 旧记录。

### 2.3 approvals 表扩展

新增 gate_level=1 的行。新增一个列用于存储 Gate1 特有的 A3 返工标记：

```sql
ALTER TABLE approvals ADD COLUMN a3_rework BOOLEAN DEFAULT false;
-- 仅 gate_level=1 时有效，gate_level=0 时始终为 false
```

> `a3_rework` 是 DB 持久化列（非仅 NATS payload 字段），用于审计查询"Gate1 是否要求了 A3 返工"。

### 2.4 event_log 表

无需变更。

## 三、新增表

### 3.1 prototype_artifacts 表（A3 管理）

```sql
CREATE TABLE prototype_artifacts (
    id              BIGSERIAL PRIMARY KEY,
    req_id          UUID NOT NULL REFERENCES requirements(id),
    cycle           INT NOT NULL DEFAULT 0,
    version         INT NOT NULL DEFAULT 1,        -- 原型版本号（标注后递增）
    prototype_url   TEXT,                           -- S3/OSS 存储 URL
    html_content    TEXT,                           -- HTML 原型源码（S3 不可用时的 fallback）
    screens         JSONB,                          -- [{name, description, url}]
    annotations     JSONB DEFAULT '[]'::jsonb,      -- 用户标注
    status          VARCHAR(20) DEFAULT 'draft',    -- 'draft' | 'confirmed'
    created_at      TIMESTAMPTZ DEFAULT NOW(),
    updated_at      TIMESTAMPTZ DEFAULT NOW(),

    UNIQUE (req_id, cycle, version)
);

CREATE INDEX idx_prototype_artifacts_req ON prototype_artifacts(req_id, cycle, version DESC);
```

**annotations 结构（JSONB 数组）：**

```json
[
  {
    "annotation_id": "uuid",
    "element_id": "#user-table-header",
    "type": "layout_change",
    "comment": "表格列宽需要调整为三列等宽",
    "position": {"x": 120, "y": 45},
    "created_at": "ISO 8601"
  }
]
```

**annotation type 枚举：**

| type | 含义 |
|------|------|
| `layout_change` | 布局调整 |
| `content_change` | 内容修改 |
| `style_change` | 样式调整 |
| `add_element` | 新增元素 |
| `remove_element` | 删除元素 |
| `flow_change` | 交互流程调整 |
| `other` | 其他 |

**screens 结构（JSONB 数组）：**

```json
[
  {
    "name": "列表页-默认状态",
    "description": "展示用户列表，含搜索和分页",
    "url": "https://s3/xxx/screen_1.png",
    "state": "default"
  }
]
```

**screen state 枚举：** `default` | `loading` | `empty` | `error` | `hover` | `active`

### 3.2 design_specs 表（A4 管理）

```sql
CREATE TABLE design_specs (
    id              BIGSERIAL PRIMARY KEY,
    req_id          UUID NOT NULL REFERENCES requirements(id),
    cycle           INT NOT NULL DEFAULT 0,
    version         INT NOT NULL DEFAULT 1,        -- 修订版本号（Gate1 打回后 A4 重出时递增）
    spec_doc        JSONB,                          -- 结构化 Spec 文档
    openapi_schema  JSONB,                          -- OpenAPI 3.0 Schema
    erd_diagram     JSONB,                          -- ERD 实体关系图（结构化 JSON）
    ddl_statements  TEXT,                           -- DDL SQL 语句
    quality_score   NUMERIC(3,2) CHECK (quality_score >= 0 AND quality_score <= 1),
    created_at      TIMESTAMPTZ DEFAULT NOW(),
    updated_at      TIMESTAMPTZ DEFAULT NOW(),

    UNIQUE (req_id, cycle, version)
);

CREATE INDEX idx_design_specs_req ON design_specs(req_id, cycle, version DESC);
```

```json
  "annotation_count": 5
```

### 4.3 A3 确认时的 DB 写入事务

```
BEGIN
  1. UPDATE prototype_artifacts SET status='confirmed' WHERE req_id=? AND cycle=? AND version=(SELECT MAX(version))
  2. INSERT INTO agent_results (req_id, agent_key='A3', cycle, status='completed', artifact)
     ON CONFLICT (req_id, agent_key, cycle) DO UPDATE
       SET artifact=EXCLUDED.artifact, status='completed', created_at=NOW()
  3. UPDATE requirements SET design_status='spec_writing'
  4. 发布 NATS: agent.result.A3
COMMIT
```

```json
{
  "check_report": {
    "overall_score": 0.78,

      {
        "dimension": "api_consistency",
```

### 6.3 检查维度枚举

| dimension | 含义 | 检查内容 |
|-----------|------|---------|
| `api_consistency` | API 一致性 | OpenAPI Schema 与 Spec 文档中的 API 定义是否一致 |
| `erd_completeness` | ERD 完整性 | ERD 是否覆盖 Spec 中所有业务实体，字段类型是否完整 |
| `state_machine_closure` | 状态机闭合性 | Spec 中的状态机是否所有状态都有入口/出口 |
| `prototype_spec_alignment` | 原型-Spec 对齐 | 原型页面是否覆盖 Spec 中所有用例的交互路径 |
| `security_baseline` | 安全基线 | API 是否有认证/授权定义，敏感字段是否标注 PII |

### 6.4 问题严重度枚举

| severity | 含义 | 示例 |
|----------|------|------|
| `critical` | 致命 | ERD 缺少核心业务实体 |
| `major` | 主要 | API 路径与 Spec 不一致 |
| `minor` | 次要 | 字段缺少注释 |
| `info` | 提示 | 可选的性能优化建议 |

## 七、Gate1 数据规范

### 7.1 产物存储

Gate1 **自行写入**审批记录到 `approvals` 表（gate_level=1）。Orchestrator 仅订阅 `agent.result.gate1.*` 获取决策并据此编排下一动作。

### 7.2 审批记录结构

```json
{
  "id": "uuid",
  "req_id": "uuid",
  "session_id": "uuid",
  "gate_level": 1,
  "cycle": 0,
  "decision": "pass | reject",
  "reject_reasons": [
    {
      "category": "spec_incomplete",
      "description": "用户管理模块的权限校验流程未在 Spec 中描述"
    }
  ],
  "revision_guidance": "请补充权限校验相关的状态机流转和 API 定义",
  "a3_rework": false,
  "reviewer_user_id": "string",
  "reviewer_name": "string",
  "reviewed_at": "ISO 8601"
}
```

### 7.3 拒绝原因枚举

| category | 含义 | UI 标签（前端展示） |
|----------|------|-------------------|
| `prototype_not_aligned` | 原型与需求草案不一致 | 原型与需求不符 |
| `spec_incomplete` | Spec 文档不完整，缺少关键模块/流程 | Spec 不完整 |
| `api_design_issue` | API 设计有缺陷（接口、参数、响应） | API 设计问题 |
| `erd_incomplete` | ERD 缺少关键实体或字段 | ERD 不完整 |
| `acceptance_criteria_mismatch` | 设计未覆盖所有验收标准 | 验收标准遗漏 |
| `prototype_change_needed` | 原型需要修改（触发 A3 返工） | 需要原型修改 |
| `other` | 其他原因 | 其他 |

> **`prototype_change_needed` 的特殊处理：** 审批人勾选此项时，`a3_rework` 字段设为 `true`。Orchestrator 收到后发布 `context.ready.A3` 而非 `context.ready.A4`，A3 修订确认后重新走 A4→A5→Gate1 链路。**此项不改变 cycle 计数器**，使用 `design_revision_count` 追踪。

### 7.4 NATS 事件

#### context.ready.gate1（Orchestrator → Gate1）

```json
{
  "req_id": "uuid",
```

## 八、全链路事件流（阶段二）

### 8.1 正常流程

```
Gate0 pass → Orchestrator:
  → 写入 event_log (direction='IN', event_name='agent.result.gate0.pass')
  → 更新 requirements: phase='design', design_status='prototyping'
  → 查询 DB（MAX cycle）: requirements + agent_results (A1, A2)
  → build_context → 发布 NATS: context.ready.A3
  → 写入 event_log (direction='OUT', event_name='context.ready.A3')

A3 原型生成（HTTP+SSE 多轮对话）
  → 用户确认 → POST /api/prototype/confirm
  → 持久化:
      - prototype_artifacts (status='confirmed')
      - agent_results (agent_key='A3', cycle, status='completed')
      - requirements.design_status = 'spec_writing'
  → 发布 NATS: agent.result.A3

Orchestrator 收到 agent.result.A3:
  → 写入 event_log (direction='IN', event_name='agent.result.A3')
  → 查询 DB:
      - requirements.requirement_draft + agent_results A1: a1_output
      - agent_results A2: a2_output
      - prototype_artifacts (status='confirmed'): a3_output
  → build_context → 发布 NATS: context.ready.A4
  → 写入 event_log (direction='OUT', event_name='context.ready.A4')

A4 Spec 撰写:
  → 执行 Spec/OpenAPI/ERD/DDL 生成
  → 持久化:
      - design_specs (INSERT, version=1 或 MAX(version)+1)
      - agent_results (agent_key='A4', cycle, status='completed') — UPSERT
  → 发布 NATS: agent.result.A4

Orchestrator 收到 agent.result.A4:
  → 写入 event_log (direction='IN', event_name='agent.result.A4')
  → 更新 requirements.design_status = 'design_checking'
  → 查询 DB: a3_output + a4_output (design_specs)
  → build_context → 发布 NATS: context.ready.A5
  → 写入 event_log (direction='OUT', event_name='context.ready.A5')

A5 设计检查:
  → 执行五维检查
  → 持久化: agent_results (agent_key='A5', cycle, status='completed')
  → 发布 NATS: agent.result.A5

Orchestrator 收到 agent.result.A5:
  → 写入 event_log (direction='IN', event_name='agent.result.A5')
  → 更新 requirements.design_status = 'design_completed'
  → 查询 DB: a1_output + a2_output + a3_output + a4_output + a5_output
  → build_context → 发布 NATS: context.ready.gate1
  → 写入 event_log (direction='OUT', event_name='context.ready.gate1')

Gate1 审批:
  → MC Backend 收到 context.ready.gate1 → 预创建 approvals (gate_level=1, status='pending')
  → 审批人决策:
      ├── ✅ pass:
      │     → Gate1 更新 approvals: decision='pass', reviewer_*
      │     → 发布 agent.result.gate1.pass
      │     → Orchestrator: event_log → requirements.phase='tech_prep' → 启动阶段三（A6+A7）
      └── ❌ reject:
            → Gate1 更新 approvals: decision='reject', reject_reasons, revision_guidance, a3_rework
            → 发布 agent.result.gate1.reject → 进入打回链路
```

### 8.2 Gate1 打回链路

```
Orchestrator 收到 agent.result.gate1.reject:
  → 写入 event_log (direction='IN', event_name='agent.result.gate1.reject')
  → 更新 requirements:
      - design_status = 'gate1_rejected'
      - design_revision_count += 1
  → 判断 a3_rework 标记:
      ├── a3_rework = false（默认）:
      │     → 查询 DB: a1_output + a2_output + a3_output + a5_output + gate1_rejection
      │     → build_context（含 revision_context.is_revision=true）
      │     → 发布 context.ready.A4
      │     → requirements.design_status = 'spec_writing'
      └── a3_rework = true:
            → 查询 DB: a1_output + a2_output + gate1_rejection
            → build_context（含 revision_context）
            → 发布 context.ready.A3
            → requirements.design_status = 'prototyping'

修订后完整重走 A3→A4→A5→Gate1 或 A4→A5→Gate1 链路（cycle 不变）
```

### 8.3 A4/A5 超时降级

```
A4 超时（10 分钟）:
  → Orchestrator 重试 1 次 → 仍超时
  → 写入 agent_results (agent_key='A4', cycle, status='skipped')
  → a4_output = {a4_missing: true, 其余字段为空}
  → 发布 context.ready.A5（A5 仅基于 A3 原型做检查）
  → A5 check_report 中标注 a4_missing

A5 超时（10 分钟）:
  → Orchestrator 重试 1 次 → 仍超时
  → 写入 agent_results (agent_key='A5', cycle, status='skipped')
  → a5_output = {a5_missing: true}
  → 发布 context.ready.gate1（无检查报告，审批人仅看 A3+A4 产物）
```

## 九、审计查询速查（阶段二）

| 审计问题 | 查询路径 |
|---------|---------|
| 原型经过几次标注才确认？ | `SELECT MAX(version) FROM prototype_artifacts WHERE req_id=? AND status='confirmed'` |
| 第N轮 Spec 完整内容？ | `SELECT * FROM design_specs WHERE req_id=? AND cycle=N ORDER BY version DESC LIMIT 1` |
| A4 被 Gate1 打回了几次？ | `requirements.design_revision_count` WHERE req_id=? |
| A5 检查报告细节？ | `agent_results WHERE agent_key='A5' AND cycle=? → artifact.check_report` |
| Gate1 是否要求了 A3 返工？ | `approvals WHERE gate_level=1 AND req_id=? → a3_rework` 字段 |
| 阶段二耗时？ | `event_log WHERE req_id=? AND event_name IN ('context.ready.A3', 'agent.result.gate1.pass')` 的首末 created_at 差值 |

## 十、数据库定位速查（阶段二）

| 谁需要什么数据 | 从哪取 |
|--------------|--------|
| Orchestrator 需要 session_id/cycle | NATS 事件 payload 直接取 |
| A3 需要需求草案 | `context.ready.A3` payload（Orchestrator 从 `requirements` + `agent_results` A1 查询） |
| A4 需要 A1+A2+A3 产物 | `context.ready.A4` payload |
| A5 需要 A3+A4 产物 | `context.ready.A5` payload |
| Gate1 需要全部产物 | `context.ready.gate1` payload |
| A4 修订需要 A5 报告 + 拒绝原因 | `context.ready.A4` payload 中 revision_context |
| A3 返工需要拒绝原因 | `context.ready.A3` payload（含 gate1_rejection） |
| 最新原型 | `prototype_artifacts` WHERE req_id=? AND status='confirmed' |
| 最新 Spec | `design_specs` WHERE req_id=? AND cycle=(SELECT MAX(cycle)) |
