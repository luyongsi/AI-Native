# A1 多轮对话存储与状态流转方案

## 问题背景

参考豆包、DeepSeek等网页端对话模式，A1需要支持：
1. 多轮对话数据的缓存和持久化存储
2. 用户级别隔离（不同用户看到不同的对话历史）
3. LLM调用、MCP工具调用的完整审计链路
4. 对话确认后与Orchestrator状态机的集成
5. 人工门禁打回后的状态流转

---

## 方案概览

### 整体架构

用户浏览器 (WebSocket) → MC Backend (FastAPI) → PostgreSQL + Redis → NATS → A1 Agent ↔ Orchestrator

### 数据流转路径

1. 用户输入 → WebSocket → MC Backend → 写DB → 发NATS → A1处理
2. AI理解 → A1 → 写DB → 发NATS → MC Backend → WebSocket → 用户
3. 用户确认 → WebSocket → MC Backend → 写DB → 发NATS → Orchestrator Signal
4. Gate打回 → Orchestrator → 发NATS → MC Backend → 写DB → WebSocket → 用户
