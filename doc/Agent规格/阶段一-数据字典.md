## 一、核心原则

### 1.1 产物持久化

| 组件 | 写入职责 |
|------|---------|
| **A1** | `requirements` 表（首次创建 req_id / 确认时更新草案和元数据）、`agent_results` (A1)、`dialogue_sessions`、`dialogue_messages`、`understanding_snapshots` |
| **A2** | `agent_results` (A2) |
| **Gate0** | `approvals` 表（MC Backend 预创建 `status='pending'` 行，Gate0 在审批人提交后更新决策，不依赖 Orchestrator） |
| **Orchestrator** | `event_log` 表（每收发一条 NATS 事件追加一行）+ `requirements` 表编排字段（status / gate_rejection_count）+ `dialogue_sessions` 重激活（status='reopened'） |

### 1.2 req_id 创建时机

```
用户点击"创建需求" → POST /api/requirements → 创建 requirements 行 (status='draft')
    → 返回 req_id
用户进入对话页面 → POST /api/dialogue/chat {req_id, message}
    → MC Backend 创建 dialogue_sessions (req_id 已有) + 首条 dialogue_messages
    → A1 LLM 分析 → 多轮对话（HTTP+SSE）
用户确认 → POST /api/dialogue/confirm → 更新 requirements + 写入 agent_results
```

- `req_id` 在对话前由用户主动创建，贯穿全链路
- `session_id` 在首次对话时由 MC Backend 创建

### 1.3 多轮循环（Cycle）

Gate0 拒绝后 A1→A2→Gate0 完整重走。`cycle` 是**唯一轮次标识**，替代了旧的 `reopen_count`/`reopen_round` 概念：

```
cycle = 0: req_id 创建 + A1 首次对话 → A2 首次分析 → Gate0 首次审批
cycle = 1: Gate0 拒绝 → A1 修订 → A2 重分析 → Gate0 再次审批
cycle = 2: ...
```

- `cycle` = `requirements.gate_rejection_count`（首次为 0）
- 所有 NATS 事件 payload **必须**携带 `cycle`
- `dialogue_messages.cycle` — 标识消息属于哪一轮
- `understanding_snapshots.cycle` — 标识快照属于哪一轮
- `agent_results` 通过 `(req_id, agent_key, cycle)` 约束保留每轮版本
- 前端按 `cycle` 分组展示对话历史

### 1.4 Orchestrator 的数据获取

| 用途 | 数据来源 |
|------|---------|
| **路由决策**（下一步触发谁、session_id） | NATS 事件 payload 直接取，不查 DB |
| **build_context**（为下游组装上下文） | 查询 `requirements` 表 + `agent_results` 表（MAX cycle） |
| **cycle 递增** | Gate0 拒绝时 `requirements.gate_rejection_count += 1` |
| **事件审计** | 每收发一条 NATS 事件写入 `event_log` 一行 |

### 1.5 字段命名规则

- 全链路统一字段名，禁止同一概念有多个名称
- JSON 字段使用 `snake_case`
- 下文定义的字段名、类型、枚举值是**规范**，所有文档必须对齐

## 二、全局路由键

全链路传递，不可丢失：

| 字段 | 类型 | 说明 | 创建者 |
|------|------|------|--------|
| `req_id` | `UUID` | 需求唯一标识 | MC Backend（用户点击"创建需求"） |
| `session_id` | `UUID` | 对话会话唯一标识 | MC Backend（首次 POST /api/dialogue/chat） |
| `cycle` | `int` | 当前轮次，由 `requirements.gate_rejection_count` 派生，首次 = 0（MC Backend 建表 DEFAULT 0） | Orchestrator（仅 Gate0 拒绝时递增 `gate_rejection_count`） |

> 所有 NATS 事件 payload **必须**携带这三个字段。

## 三、数据库表

### 3.1 requirements 表（需求主表）

```sql
CREATE TABLE requirements (
    id                  UUID PRIMARY KEY,
    status              VARCHAR(50) NOT NULL,
    -- A1 范围内: 'draft' → 'analyzing_completed' ↔ 'gate_rejected'
    -- Gate0 通过后: 'approved'（Orchestrator 更新）

    requirement_draft   JSONB,          -- 最新版需求草案（镜像，仅保留最新）
    confidence_score    NUMERIC(3,2),   -- A1 置信度评分

    creator_user_id     VARCHAR(255),
    creator_name        VARCHAR(255),
    analyzer_agent      VARCHAR(50) DEFAULT 'A1',
    analyzed_at         TIMESTAMPTZ,

    gate_rejection_count INT DEFAULT 0,         -- 同时也是 cycle 计数器
    last_gate_rejection  JSONB,                 -- 最新一次打回信息
    revision_count       INT DEFAULT 0,
    last_revised_at      TIMESTAMPTZ,

    created_at          TIMESTAMPTZ DEFAULT NOW(),
    updated_at          TIMESTAMPTZ DEFAULT NOW()
);
```

> `requirement_draft` 为最新版镜像（方便直接查当前态）。历史版本通过 `agent_results WHERE agent_key='A1'` 按 cycle 查询。

### 3.2 agent_results 表（Agent 产物，独立表）

```sql
CREATE TABLE agent_results (
    id          BIGSERIAL PRIMARY KEY,
    req_id      UUID NOT NULL REFERENCES requirements(id),
    agent_key   VARCHAR(10) NOT NULL,    -- 'A1', 'A2', ...
    cycle       INT NOT NULL DEFAULT 0,
    status      VARCHAR(20) NOT NULL DEFAULT 'completed',
    -- 'completed': 正常完成，有有效产物（A1 始终用此值；A2 正常产出/启发式降级时也用此值）
    -- 'empty':    执行完成但完全无匹配结果（仅 A2：MCP 全部失败且无启发式替代时）
    -- 'skipped':  被 Orchestrator 跳过（仅 A2 超时降级），由 Orchestrator 写入
    artifact    JSONB NOT NULL,
    created_at  TIMESTAMPTZ DEFAULT NOW(),

    UNIQUE (req_id, agent_key, cycle)
);

CREATE INDEX idx_agent_results_req ON agent_results(req_id, agent_key, cycle DESC);
```

> `status` 枚举适用性：A1 始终为 `'completed'`；A2 正常产出/有启发式产物时为 `'completed'`，MCP 全部失败且完全无产出时为 `'empty'`，被 Orchestrator 超时跳过时为 `'skipped'`（由 Orchestrator 写入）。

### 3.3 event_log 表（Orchestrator 事件审计）

```sql
CREATE TABLE event_log (
    id          BIGSERIAL PRIMARY KEY,
    req_id      UUID,
    session_id  UUID,
    cycle       INT,
    event_name  VARCHAR(100) NOT NULL,   -- 'agent.result.A1', 'context.ready.A2', ...
    direction   VARCHAR(10) NOT NULL,    -- 'IN' 接收 / 'OUT' 发布
    payload     JSONB NOT NULL,          -- 完整事件 payload
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_event_log_req ON event_log(req_id, cycle);
CREATE INDEX idx_event_log_name ON event_log(event_name);
```

Orchestrator 每收发一条 NATS 事件追加一行，不遗漏。事后可完整重建事件时间线。

### 3.4 approvals 表（Gate0 写入）

```sql
CREATE TABLE approvals (
    id              UUID PRIMARY KEY,
    req_id          UUID NOT NULL REFERENCES requirements(id),
    session_id      UUID NOT NULL,
    gate_level      INT NOT NULL DEFAULT 0,
    cycle           INT NOT NULL,
    status          VARCHAR(20) NOT NULL DEFAULT 'pending',    -- 'pending' → 'decided'
    decision        VARCHAR(10),                               -- 'pass' / 'reject'（status='decided' 时写入）
    reject_reasons  JSONB,                      -- [{category, description}]
    revision_guidance TEXT,                     -- 修订指引（reject 时必填）
    reviewer_user_id  VARCHAR(255),
    reviewer_name     VARCHAR(255),
    reviewed_at     TIMESTAMPTZ,
    created_at      TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_approvals_req ON approvals(req_id, cycle);
```

> `approvals` 的写入分为两步：
> 1. **预创建**：MC Backend 收到 `context.ready.gate0` 时，在 `approvals` 表插入一行 `status='pending'`，生成 `approval_id`。此 ID 用于后续审批 API 的路由。
> 2. **决策写入**：审批人提交决策后，Gate0 更新该行：`status='decided'`, `decision='pass'|'reject'`, `reviewer_*`, `reviewed_at`。
>
> Orchestrator 订阅 `agent.result.gate0.*` 获取决策结果，但不负责持久化 approvals。

### 3.5 dialogue_sessions 表（A1 管理）

```sql
CREATE TABLE dialogue_sessions (
    id                  UUID PRIMARY KEY,
    req_id              UUID UNIQUE REFERENCES requirements(id) ON DELETE CASCADE,
    status              VARCHAR(50) DEFAULT 'active',
    -- 'active' | 'completed' | 'reopened' | 'abandoned'

    iterations          INT DEFAULT 0,
    total_messages      INT DEFAULT 0,
    current_understanding JSONB,
    clarification_points  JSONB,
    confidence_score    NUMERIC(3,2),
    human_confirmations JSONB DEFAULT '[]'::jsonb,

    creator_user_id     VARCHAR(255),
    creator_name        VARCHAR(255),

    created_at          TIMESTAMPTZ DEFAULT NOW(),
    last_updated        TIMESTAMPTZ DEFAULT NOW(),
    first_confirmed_at  TIMESTAMPTZ,
    last_confirmed_at   TIMESTAMPTZ
);
```

> `req_id` ↔ `dialogue_sessions` 为 **1:1** 关系：一个需求只有一个对话会话，修订在原会话中继续。
### 3.6 dialogue_messages 表（A1 管理）

```sql
CREATE TABLE dialogue_messages (
    id                      BIGSERIAL PRIMARY KEY,
    session_id              UUID REFERENCES dialogue_sessions(id) ON DELETE CASCADE,
    role                    VARCHAR(20) CHECK (role IN ('human', 'ai', 'system')),
    content                 JSONB NOT NULL,
    cycle                   INT DEFAULT 0,          -- 所属轮次（替代旧 reopen_round）
    understanding_snapshot_id BIGINT,
    timestamp               TIMESTAMPTZ DEFAULT NOW(),
    sequence_number         INT NOT NULL
);
```

`content` JSONB 结构按 `role` 区分：

| role | content 结构 |
|------|-------------|
| `human` | `{"text": "用户原始消息"}` |
| `ai` | `{"text": "A1 回复文本", "draft_preview": {...}, "clarifications": [...]}` |
| `system` | `{"type": "gate_rejection", "reject_reasons": [...], "revision_guidance": "...", "cycle": 0}` |

### 3.7 understanding_snapshots 表（A1 管理）

```sql
CREATE TABLE understanding_snapshots (
    id                  BIGSERIAL PRIMARY KEY,
    session_id          UUID REFERENCES dialogue_sessions(id) ON DELETE CASCADE,
    iteration           INT NOT NULL,
    cycle               INT DEFAULT 0,          -- 所属轮次
    draft               JSONB NOT NULL,
    clarification_points JSONB,
    confidence_score    NUMERIC(3,2),
    knowledge_sources   JSONB,
    mcp_tools_used      JSONB,
    created_at          TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_understanding_snapshots_session_cycle ON understanding_snapshots(session_id, cycle);
```

### 3.8 human_confirmations 结构

`dialogue_sessions.human_confirmations` (JSONB 数组)：

```json
[
  {
    "confirmed_at": "ISO 8601",
    "cycle": 0,
    "final_notes": "用户补充说明（可选）"
  }
]
```

每次用户确认时追加一条，按 `confirmed_at` 排序即为确认历史。

## 七、全链路事件流

### 7.1 正常流程

```
0. 用户点击"创建需求" → POST /api/requirements
   → MC Backend 创建 requirements (status='draft', gate_rejection_count=0)
   → 返回 req_id

1. 用户进入对话页面 → POST /api/dialogue/chat {req_id, message}
   → MC Backend 创建 dialogue_sessions + 首条 dialogue_messages (cycle=0)

2. A1 LLM 分析 → 多轮对话（HTTP+SSE）
   → 每次迭代: understanding_snapshots (cycle=0) + dialogue_messages (cycle=0)

3. 用户 POST /api/dialogue/confirm
   → 持久化:
      - requirements.requirement_draft = 最终草案（最新镜像）
      - requirements.confidence_score = 置信度
      - requirements.status = 'analyzing_completed'
      - requirements.analyzed_at = NOW()
      - requirements.revision_count += 1（修订时）
      - requirements.last_revised_at = NOW()（修订时）
      - agent_results (agent_key='A1', cycle, status='completed'):
        artifact = {requirement_draft: <该轮完整草案>, wireframe_url: ...}
      - dialogue_sessions.status = 'completed'
      - dialogue_sessions.human_confirmations 追加一条确认记录
   → 发布 NATS: agent.result.A1 (cycle = gate_rejection_count)

4. Orchestrator:
   → 写入 event_log (direction='IN', event_name='agent.result.A1')
   → 提取 req_id, session_id, cycle（路由用）
   → 查询 DB（MAX cycle）:
      - requirements 表: requirement_draft（最新镜像）, confidence_score
      - agent_results WHERE agent_key='A1' AND cycle=<当前>: wireframe_url
   → build_context → 发布 NATS: context.ready.A2
   → 写入 event_log (direction='OUT', event_name='context.ready.A2')

5. A2 收到 context.ready.A2
   → 执行分析（MCP 知识库检索）
   → 持久化: agent_results (agent_key='A2', cycle, status)
   → 发布 NATS: agent.result.A2

6. Orchestrator:
   → 写入 event_log (direction='IN', event_name='agent.result.A2')
   → 提取 cycle（路由）
   → 查询 DB（MAX cycle）:
      - requirements.requirement_draft + agent_results A1: a1_output
      - agent_results WHERE agent_key='A2': a2_output
      - 若 A2 被跳过: a2_output.a2_missing = true（Orchestrator 写入 agent_results A2 status='skipped'）
   → build_context → 发布 NATS: context.ready.gate0
   → 写入 event_log (direction='OUT', event_name='context.ready.gate0')

7. Gate0 审批人决策
   → 通过:
      - Gate0 写入 approvals (decision='pass')
      - 发布 agent.result.gate0.pass
      - Orchestrator: event_log → requirements.status = 'approved' → 启动阶段二（A3）
   → 拒绝:
      - Gate0 写入 approvals (decision='reject')
      - 发布 agent.result.gate0.reject → 进入打回链路
```

### 7.2 Gate0 打回链路

```
8. Orchestrator 收到 agent.result.gate0.reject
   → 写入 event_log (direction='IN', event_name='agent.result.gate0.reject')
   → 从 payload 中直接取 session_id、cycle（不查 DB）
   → 更新 requirements:
      - status = 'gate_rejected'
      - gate_rejection_count += 1   ← cycle 递增
      - last_gate_rejection = reject 信息
   → 更新 dialogue_sessions: status = 'reopened'
   → 注入 dialogue_messages: role='system', type='gate_rejection', cycle = 旧 cycle
   → 发布 context.ready.A1 (cycle = gate_rejection_count 新值)
   → 写入 event_log (direction='OUT', event_name='context.ready.A1')

9. A1 收到 context.ready.A1
   → 会话 REOPENED
   → 用户看到打回原因，继续对话（HTTP+SSE）
   → 新消息 cycle = gate_rejection_count 新值
   → 修订完成 → POST /api/dialogue/confirm
   → 发布 agent.result.A1 (cycle = gate_rejection_count 新值)

10. 返回步骤 4，完整重走 A2 → Gate0（cycle 递增）
```

### 7.3 A2 超时降级链路

```
5a. Orchestrator:
    → A2 执行超时 → 重试 1 次 → 仍超时
    → 写入 agent_results (agent_key='A2', cycle, status='skipped')
    → 查询 DB: requirements.requirement_draft + agent_results A1
    → a2_output = {a2_missing: true, 其余字段为空}
    → 发布 context.ready.gate0
```

## 八、审计查询速查

| 审计问题 | 查询路径 |
|---------|---------|
| 需求经历了哪些轮次？ | `SELECT DISTINCT cycle FROM agent_results WHERE req_id=?` + `SELECT * FROM approvals WHERE req_id=?` |
| 第N轮 A1 草案完整内容？ | `agent_results WHERE agent_key='A1' AND cycle=N → artifact.requirement_draft` |
| 第N轮 A2 产物是什么？ | `agent_results WHERE agent_key='A2' AND cycle=N → artifact` |
| 用户何时确认的？ | `dialogue_sessions.human_confirmations` 按 `confirmed_at` 排序 |
| 两次修订间草案改了什么？ | 取 cycle=N 和 cycle=N+1 的 A1 artifact.requirement_draft 做 diff |
| 需求从创建到 Gate0 通过花了多久？ | `requirements.created_at` → `approvals WHERE decision='pass' → reviewed_at` |
| 完整事件时间线？ | `event_log WHERE req_id=? ORDER BY created_at` |
| A2 是否被跳过？ | `agent_results WHERE agent_key='A2' AND status='skipped'` |

## 九、数据库定位速查

| 谁需要什么数据 | 从哪取 |
|--------------|--------|
| Orchestrator 需要 session_id | NATS 事件 payload 直接取 |
| Orchestrator 需要 cycle | NATS 事件 payload 直接取 |
| A2 需要 A1 需求草案 | `context.ready.A2` payload（Orchestrator 从 `requirements` + `agent_results` A1 查询后组装） |
| Gate0 需要 A1 产出 | `context.ready.gate0` payload（Orchestrator 从 `requirements` + `agent_results` A1 查询） |
| Gate0 需要 A2 产出 | `context.ready.gate0` payload（Orchestrator 从 `agent_results` A2 查询） |
| 打回时找 A1 会话 | `agent.result.gate0.reject` payload 中 `session_id`，Orchestrator 直接透传 |
| A1 修订需要历史对话 | `dialogue_messages` WHERE session_id=? AND cycle=? |
| A1 修订需要拒绝原因 | `context.ready.A1` payload（NATS 直传） |
| 最新轮次产物 | `agent_results` WHERE req_id=? AND cycle=(SELECT MAX(cycle)) |
| 事件审计 | `event_log` WHERE req_id=? ORDER BY created_at |
