## 一、产品理念：人机协同的三个原则

| 原则 | 含义 | 在产品上的体现 |
|---|---|---|
| **看图提需求，而非读字** | 需求阶段就把抽象文字变成可点的低保真原型与验收清单，消除"PRD 与实现不符"的歧义 | 需求录入工作台的"原型面板"实时渲染 |
| **AI 干活全程透明** | 人随时能看到"哪个 Agent 在干什么、卡在哪、要不要我拍板" | Mission Control 的活动直播、协作拓扑、审批中心 |
| **人只在关键点拍板** | AI 负责 ~80% 确定性执行，人聚焦 ~20% 的决策；门禁有 SLA，不让人被流程淹没 | Gate 0–3 审批 + 超时升级 + 个人工作台"优先处理" |

> 与 `02` 的对应：本产品是控制面与执行面的"人机协同界面（Mission Control）"，把 Agent 的状态、产物、循环、门禁投射给人，并把人的决策回灌给系统。

## 二、前道：需求录入与原型生成

### 2.1 多源需求汇入（四种来源）

需求不再只靠手敲，而是从团队真实工作流里被动/主动捕获：

| 来源 | 方式 | 交互要点 |
|---|---|---|
| **飞书群聊**（被动） | @PRD Bot 或语义识别潜在需求 | Bot 回飞书卡片，给 **3–5 个快速确认选项**（导出范围/单次上限/权限控制），勾选即建草稿 |
| **飞书会议**（主动） | 会议录制结束自动转写 + 提取 | 智能纪要列出多个潜在需求 + 关键发言时间戳，一键"创建需求草稿" |
| **飞书文档**（主动拉取） | 粘贴文档链接自动解析 | 解析内容标记状态 ✅已识别 / ⚠️存疑 / ⏳处理中 |
| **手动创建** | Mission Control 直接新建 | 标准表单入口 |

- **冲突检测**：多源同步时自动标记 ⚠️ 冲突项，由产品经理点击 [查看]/[已解决]。
- **来源可追溯**：右侧"信息来源"面板可跳转原聊天、播放会议关键片段、打开原文档。

### 2.2 三栏需求工作台

布局比例 **30% 对话 ｜ 40% Spec ｜ 30% 原型**；响应式断点：≥1440px 三栏 / 1024–1439px 两栏+底部抽屉 / <1024px 单栏标签页。

```
┌────────────┬──────────────────────┬────────────────┐
│ 💬 对话      │ 📄 Spec（渐进生长）     │ 🖼️ 原型         │
│ PRD Agent   │ 章节导航 + 完成度评分    │ 沙箱实时渲染     │
│ 多轮澄清     │ 实时编辑/确认           │ 交互状态切换     │
│ 用户输入     │                        │ 设备预览/标注    │
└────────────┴──────────────────────┴────────────────┘
```

### 2.3 对话面板：三种交互模式

PRD Agent 的对话遵循状态机：`Idle → Understanding → Generating → WaitingForUser`（外部事件可触发 `Notifying` 刷新）。三种模式：

- **模式 A · 选择题**：Agent 给 **≤5 个选项**（超出折叠为"其他方案…"），用户点选 + 可参考其他项目做法。
- **模式 B · 开放追问**：Agent 列缺口清单，用户自然语言补充或点 [AI 帮我补充]。
- **模式 C · 一键确认**：Agent 自动补全边界条件（空/加载/错误/权限），用户 [全部接受]/[逐条审核]/[撤回]。
- `WaitingForUser` 超过 **30min** 自动挂起提醒。

### 2.4 Spec 面板：渐进式结构化生长

Spec 不是一次写完，而是随对话"长"出来。章节状态流：

```
⏳待补充 → 🟡部分 → ✅已补全 ↔ 🔵已修改(需重新确认)
                ↑(审批打回 🔴)        │(重新确认)
                └─────────────────────┘
```

**完成度评分**：各章节完成度 × 权重，实时显示；**阈值 90% 即可提交评审**。示例：

```
📊 Spec 完整度: 85%   ████████████████░░░░
✅ 功能概述 100% │ ✅ 用户故事 100% │ 🟡 验收标准 75% │ ⏳ 接口契约 0%
💡 建议: 补充"导出超时上限""最大文件大小"两条验收
```

### 2.5 原型面板：生成、标注、版本

- **生成**：从 Spec 自动出可交互原型；可切换 **交互状态**（默认/空数据/加载中/错误/权限受限/边界）与 **设备预览**（375 / 414 / 768 / 1440 px）。
- **标注闭环**（人机协同的典型回路）：

```
用户右键标注(位置/样式/交互/文案/组件缺失)
  → 事件 prototype.annotated
  → UI Agent 解析 + 生成代码修改
  → 沙箱热更新
  → Agent 在标注下方回复，标注状态变 ✅已解决
```

- **设计稿对比（环境锚定的视觉裁判）**：像素差异分级——`<2% 忽略 / 2–5% 建议修复 / 5–10% 默认选中 / >10% 强制选中并阻止审批`。
- **版本管理**：每次修改自动生成快照（v3→v4→v5…），支持回退与"对比两个版本"。

### 2.6 设计/需求确认门禁与打回闭环

> **门禁口径对齐**：前端规格中的"Gate 1 审批台"由产品经理确认 Spec + 原型；在 `02` 的统一四门模型里，它对应 **Gate 0（需求方向）+ Gate 1（设计）** 的合并确认点。下文沿用产品视角的"审批台"。

- **触发**：PRD Agent + UI Agent 完成初稿 → 提交评审。
- **打回结构化**：弹窗勾选 ☑Spec 需修改 / ☑边界条件不足 等；打回后对应章节标 🔴 + 原因横幅，PRD Agent 自动醒来生成针对性追问；改完点 [重新提交评审] 显示修订摘要卡（事件 `gate.resubmitted`）。
- **SLA**：产品经理 **1h 预警 / 2h 超时升级** 至产品总监 + 项目经理。

## 三、中枢：Mission Control 指挥舱

### 3.1 导航体系（角色化）

```
🏠 首页(可自定义 Widget) ├ 我的工作台 ├ 团队全景
📋 需求流   ├ 看板 ├ 列表 ├ 价值流 ├ 详情
🤖 Agent 中心 ├ 活动直播 ├ 协作拓扑 ├ 运行统计 ├ 配置
🧪 测试洞察 ├ 概览 ├ E2E 回放 ├ 视觉回归
✅ 审批中心 ├ 待我审批 ├ 我发起的 ├ 全部审批
📦 版本发布 ├ 当前进度 ├ 历史 ├ 发布日历
📊 效能仪表盘 ├ 团队效能 ├ AI 贡献度 ├ 质量趋势
🔔 告警中心 ├ 活跃告警 ├ 历史 ├ 规则配置
📁 知识库 ├ 知识图谱 ├ 文档索引 ├ 健康度
⚙️ 设置
```

**可自定义 Widget Dashboard**：按角色预设默认组合——
- 产品经理：流水线 + 我的需求 + 待审批 + Agent 活动；
- 架构师：待审方案 + 流水线 + 拓扑图 + 知识库；
- 前端：我的任务 + MR + 构建 + 相关需求变更。
支持拖拽排序、尺寸调整、独立刷新间隔、一键切换预设布局。

### 3.2 需求价值流（单需求时间轴）

把一个需求的全生命周期摊开，每阶段实际耗时对比历史基线，**超基线 50% 黄标、100% 红标**：

```
✅ 需求录入 0.5h (基线1h)   ✅ 需求澄清 1.0h (基线2h)
✅ UI原型 0.5h (基线4h)     ⏳ 设计评审 等待2.0h (SLA2h) ⚠️已超1.5h
⬜ 方案拆解 ⬜ 开发 ⬜ 测试 ⬜ 发布
统计: 已完成2.0h │ 等待2.0h │ 剩余5.0h │ AI完成度35% │ 较基线节省56%
```

### 3.3 Agent 活动直播 + 异常检测

**聚合视图**：当前活跃/空闲/异常/卡住 Agent 数；逐 Agent 显示任务、运行时长、工具调用次数、异常标记。**展开**可见完整活动时间线（think / tool_call / code_gen / test / commit / wait）。

**异常检测六规则**（对应 `02` 的异常枚举）：

| 异常 | 触发条件 | 标记 |
|---|---|---|
| 卡住 stuck | 同一状态 >5min | ⏸️ 橙 |
| 失败 failed | 任务执行错误 | ❌ 红 |
| 慢 slow | 单步 >3× 平均 | 🐢 黄 |
| 工具故障 tool_fault | 工具连续失败 3 次 | 🔧 红 |
| 超时 timeout | >预估时间 2× | ⏰ 橙 |
| 无进展 idle | 30min 无代码改动 | 😴 灰 |

每个 Agent 卡片提供 **[暂停] [介入接管] [查看完整日志] [查看 Diff]**——这是"人保持掌控"的兜底。

### 3.4 Agent 协作拓扑图

实时可视化 Orchestrator → 各 Agent 的编排关系。节点色：🟢运行 / 🔵编码 / 🟡等待 / 🔴异常 / ⬜空闲。连线三类：
- **实线+箭头 = 数据流**（含数据量，如"PRD Agent → Spec Decomposer：Spec 37KB"）；
- **虚线+箭头 = 事件触发**（"DevAgent-3 完成 → 触发 CI Agent"）；
- **点线+双向 = 资源依赖**（"DevAgent-1,2 共享 Git 仓库"）。
支持**时间回放**：拖到某时刻查看 Agent 状态快照与关键事件点。

### 3.5 测试洞察（Test Observatory）

- **总览**：总用例数、通过率、覆盖率、AI 生成占比、执行轮次。
- **分类明细**：单元/集成/E2E/视觉回归 的总数·通过·失败·来源·最后执行。
- **失败详情**：失败原因 + Agent 分析 + 修复状态（第几轮）+ [查看日志/截图/Trace]。
- **测试质量雷达**（对应 `02` 的 `qualityScore`）：边界覆盖 / 断言质量 / **变异得分** / 去重 / 稳定性，综合分 + 改进建议（如"增加并发导出场景边界测试"）。
- **E2E 回放 + 视觉回归**：Playwright Trace 时间轴 + 截图；设计稿 vs 实际渲染 + Diff 热力图（差异度 vs 阈值）。

### 3.6 审批中心（Gate 0–3）

集中处理所有人工门禁，类 PR Review 界面：左 Spec/方案、右原型/Diff、中 Agent 评审意见汇总（如 TechReviewer ✅ / UXReviewer ⚠️建议改按钮位置 / BusinessReviewer ✅）。操作：[通过并下发] / [打回修改] / [批注]。SLA 与超时升级见 §5。

### 3.7 效能仪表盘（团队 KPI）

| 指标 | 示例值 | 定义 |
|---|---|---|
| 周期时间 | 2.3 天 ↓15% | 需求提出→上线 |
| 吞吐量 | 12 个/周 ↑20% | 每周完成需求数 |
| AI 贡献度 | 68% ↑5% | AI 生成行数 / 总提交行数 |
| 代码质量分 | 92 ↑3% | MR 通过率 + 检查通过率 |
| Bug 逃逸率 | 8% ↓12% | 线上 Bug / 总 Bug |
| 瓶颈分布 | 设计评审 35% | 各阶段等待占比 |

> 注：以上为配套原型中的**示例/目标值**，落地后以真实埋点为准（与 `01` 第七章口径一致）。

### 3.8 通知、告警与全局搜索

- **通知分级**：🔴紧急(飞书+邮件+短信) / 🟡重要(飞书+站内) / 🔵普通(站内) / ⬜信息(站内折叠)。
- **告警规则**（可配）：Agent 任务失败[严重]、连续 3 次工具失败[严重]、同态 >5min[警告]、Gate 超时[警告]、版本进度落后 >15%[警告]、需求阻塞 >24h[警告]、Event Bus 积压 >1000[严重]、Worker 离线[严重]。
- **全局搜索（Ctrl+K）**：`@agent`、`#bug`、`/req`、`status:blocked`、`pm:张三`、`version:V2.3.0`、`date:` 等语法。

## 四、端到端：从需求到发布的一天

把上述能力串成一条真实时间线（人只在 4 个 ★ 点介入）：

```
09:00  销售在飞书群说"客户要订单批量导出"
09:05  PRD Agent 捕获 → 飞书卡片快速确认 → Mission Control 出现需求卡
09:30  Knowledge Analyst 可行性分析；UI Agent 出原型初稿；Spec Writer 写 PRD
10:00  Design Review Panel 自动交叉评审完成
★10:30 产品经理在审批中心点"通过"（Gate 0+1）→ 触发 Spec Decomposer
10:45  方案拆解为 DAG；Test Case Gen 同步产出测试资产
★10:50 架构师扫一眼 DAG/DB 变更，点"通过"（Gate 2）
11:00  Orchestrator 派发：后端 API/DB Migration → 云端 Codex；前端页面 → 开发者用 Cursor
11–14  并行开发；指挥舱实时显示 3 个 Agent 进度；Dev Agent 内部双脑自审
14:00  CI Agent 构建 + Auto Test Agent 执行 → 通过率 91.5%，2 个失败用例 Agent 自动修
15:00  修复循环结束(共 2 轮) 全部通过；Code Review Agent 完成全局审查
★15:30 Tech Lead 看 Diff + Review 报告，点"通过合并"（Gate 3）
16:00  Release Agent 金丝雀发布 + 监控正常 ✅；Knowledge Keeper 沉淀本次 Spec 与修复记录
```

> 对照传统流程，同一需求从"几天"压缩到"一天内"，且全程留痕、可回放、可追责——这正是"AI 干活、人掌舵"的形态。

## 六、落地证据与 MVP 分期

- **落地证据**：配套已有一套基于 Next.js 16 + React 19 的 Mission Control 原型，实现了协作拓扑、活动直播、需求看板/价值流、测试工作台、审批中心、效能仪表盘等核心视图，并配有结构化 mock 数据（多个需求、Agent、测试用例、审批项、效能 12 周趋势）。它证明本方案不止于纸面，关键交互均可运行验证。
- **产品 MVP 分期**（与 `01` 第九章对齐）：
  1. **Phase 1**：Agent 活动直播 + 测试洞察 + 审批中心（先让人"看得见、管得住"AI 的编码-测试闭环）。
  2. **Phase 2**：需求录入三栏工作台 + 原型标注闭环 + 价值流（设计左移）。
  3. **Phase 3**：效能仪表盘全量指标 + 知识库健康度 + 告警规则引擎（数据驱动持续优化）。
