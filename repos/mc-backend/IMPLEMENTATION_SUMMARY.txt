INTEGRATION POINTS
==================

1. Backend → Database
   - EmbeddingService queries knowledge_embeddings table
   - Uses asyncpg for connection pooling
   - Supports INSERT with ON CONFLICT upsert

2. Backend → Redis
   - Caches embeddings by text hash
   - TTL: 30 days
   - Fallback if Redis unavailable

3. Backend → DeepSeek API
   - Async HTTP calls via aiohttp
   - Batch requests (50 items)
   - Fallback to mock embeddings

4. Orchestrator → Backend
   - IndexingWorkflow schedules periodic scans
   - Activities pull from DB and index content
   - Stores results in knowledge_embeddings

5. Frontend ← Backend
   - POST /api/knowledge/search returns SearchResponse
   - Results include similarity scores for ranking

PERFORMANCE CHARACTERISTICS
============================

Vector Generation:
- Mock: instant (hash-based, deterministic)
- DeepSeek API: ~100ms for batch of 50
- Redis cache: ~1ms cache hit

Search:
- IVFFlat index: O(log N) with lists=100
- Similarity computation: <10ms for 1000 embeddings
- Network latency: ~5ms

Storage:
- Per vector: ~6KB (1536 dims × 4 bytes + metadata)
- 1M vectors: ~6GB storage + indexes

SECURITY CONSIDERATIONS

- Embeddings cache has no PII (text hashed)
